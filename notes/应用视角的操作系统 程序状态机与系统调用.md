## 操作系统是什么

操作系统的定义可以用一句话概括：它是负责让程序更容易运行的那层软件——让程序共享资源、与设备交互、完成各种有意思的事情。从应用视角看，操作系统就是一组 API，了解这组 API 就能用它做到 C 控制台里 printf 做不到的事。

这门课选用 C 语言而不是更高级的语言，原因很直接：像 JavaScript、TypeScript 这些语言的大量设计围绕 DOM tree 处理和类型系统，和操作系统关系不大。操作系统是硬件和软件之间的中间层，要讲操作系统就得选一个离它足够近的语言。汇编能直接和操作系统交互但没法写应用，纯自然语言又太远，C 语言恰好介于两者之间——它是整个应用生态最早被构建起来的语言，离机器最近又能写应用程序。C++ 不行，它调用虚函数时会查 vtable，编译器行为复杂得多，你很难一眼看出一行 C++ 代码会被翻译成什么指令序列；而 C 几乎总是可以把代码原样翻译成对应的汇编。

## 程序是什么：状态迁移系统

理解程序的第一步是认识到：任何 C 程序都可以改写成"每行只做一件小事"的 Simple C 形式。比如 `x = y + z + w` 可以拆成 `t1 = y + z; t2 = t1 + w; x = t2;`。条件判断里写复杂表达式？提到外面。函数调用的参数里嵌套表达式？也提出来。改写完后，每一行要么是一个赋值，要么是一个简单的条件跳转。

这和计算机系统基础课里的指令级模型完全对应：CPU 有一个状态（所有寄存器的值 + 内存的值），Program Counter（PC）指向当前指令，取出指令、执行、PC 跳到下一条。C 语言的 Simple C 形式同样如此——每条语句对应几个指令，单步执行一条语句就是一次状态迁移。分支、循环也是一样：`if` 是带条件的跳转，`while` 循环的本质也是一个带条件跳转，while 的 body 执行完后跳回条件判断处。如果把条件判断里的表达式都提取成独立的布尔变量赋值，整个程序的执行流程就和指令级的跳转完全对应了。

这就是"Everything is a state machine"的含义——它不是一句口号，而是有严格数学定义的。状态迁移系统是离散数学里的概念，用集合、关系、函数就能精确定义 C 语言的状态和状态迁移。你不一定要用集合论的语言把所有东西写出来，但要知道背后存在严格的数学理解。

### C 程序状态的精确定义

C 程序的状态不是简单地"有一个 PC"。精确地说，状态包含：一个函数调用栈，栈顶的 frame 叫 topmost frame，每个 frame 里有 PC、参数（参数也是局部变量）、局部变量；以及全局状态（全局变量的值）。程序初始时，栈里只有一个 frame——main，main 里有 argc、argv 等局部变量。执行时，每次从 topmost frame 的 PC 取出一条语句执行，PC 前进到下一条。

这里有一个关键区别：PC 不是全局唯一的，而是每个 stack frame 各自有一个 PC。执行一条语句就是从 topmost frame 的 PC 上取出语句、执行、更新那个 PC。

## 函数调用的状态迁移

函数调用是理解程序状态迁移的核心。当 topmost frame 的 PC 指向的语句是一个函数调用时，状态迁移的行为是：创建一个新的 frame 压到栈顶，新 frame 的 PC 指向被调用函数的入口，旧 frame 的 PC 更新为调用语句的下一条（return address）。所有旧的 frame 都保留不动。return 的语义就是 pop 栈顶 frame，把保存的 return address load 回 PC。

理解了这个模型，递归就不再神秘——递归调用就是不断往栈上 push frame，return 就是 pop。更重要的是，任何递归程序都可以改写成非递归的：你只需要自己维护一个栈，模拟 frame 的 push/pop，每次从栈顶 frame 的 PC 取语句执行。课程用汉诺塔做了演示——非递归版本和递归版本长得几乎一样，区别只是你手动管理栈和 PC。

这个改写之所以难，是因为大部分人没有真正理解函数调用的状态迁移。斐波那契那种 `f(n) = f(n-1) + f(n-2)` 的数学函数，调换两个调用的顺序不影响结果；但汉诺塔不行，调换顺序就输出错误。更难的是 f 和 g 互相调用的场景：f 调 g，g 又调回 f，要求写出非递归版本并打印调用序列。这个题连很多保研的研究生都写不对。有趣的是，AI 的能力在增长——GPT-3.5 能写非递归汉诺塔，GPT-4 做不对 f/g 互相调用，但到了能推理的模型（o1 系列）就能做对了。

## C 语言：一种高级的汇编语言

回到"C 是高级汇编语言"这个判断。当你把 C 改写成 Simple C 后，所有语法成分都能直接用几条显而易见的汇编指令写出来。C 的编译因此非常容易构建——基本上原样把 Simple C 翻译成汇编就行。C++ 的虚函数调用要查 vtable，编译器行为复杂；C 不存在这个问题。

有一个叫 CIL 的老项目（十几年历史，用 OCaml 写的）可以把 C99 程序编译成更简单的 C，用于程序分析。今天有大模型，可以让它帮我们把 C 翻译成 Simple C——不需要自己写编译器。课程演示了用 GDB 单步调试 Simple C 版本的程序，每条语句都只做一个赋值或一个简单操作，和指令级执行完全对应。

这也引出了"计算机世界没有魔法"的判断。所谓 call，就是往栈上 push 一个 frame（frame 的 PC 设为函数入口）；return 就是 pop 一个 frame。所有你在计算机世界里看到的东西，都是离散数学可以定义出来的东西。

## 编译器与编译正确性

讲清楚了什么是程序，就能讲什么是编译器。编译器把 `.c` 翻译成 `a.out`（广义的编译器包括汇编器、链接器），`a.out` 的执行模型是寄存器和内存上的指令执行。

早期的编译器其实就是 transpiler——做简单的语法变换，把每条语句原样翻译成指令。后来人们意识到编译器可以改变程序行为：死代码可以删掉，`x = 1 + 2 + 3` 且后面不用 x，可以直接优化掉。问题来了：编译器允许做什么优化，不允许做什么？

这引出了编译正确性的定义：如果在任何输入下，优化前后程序输出的序列（对外部可见的行为）完全相同，这个优化就是正确的。更精确地说，如果你假设 syscall 是程序与外界唯一的交互通道，那么除了 syscall 之外，所有内部的状态变化都是可以优化的——只要你对同样的输入产生同样的 syscall 序列。一个极端的例子：如果一个程序永远不向外输出任何东西且保证终止，编译器理论上可以把它优化成空程序。

不能优化的东西是什么？是编译器"看不见"的语句——对外部函数的调用。比如 printf 是在链接时才引入的外部函数，编译器不知道它的行为。它可能是汇编写的，执行它可能真的改变了物理世界（发了一个电信号出去）。所以对于外部调用，编译器必须保证传入的参数值在优化前后一致。这就解释了为什么 printf 的参数值不能被编译器随意改变——虽然从 C 语言语义看那只是一个函数调用，但编译器必须保守对待它对外输出的"电平"序列。

## 最小的 Hello World

标准 `gcc hello.c` 编译出来的程序有 70KB，objdump 一下能看到一堆你不知道干什么的代码。如果用 `gcc -c` 只编译不链接呢？会报 `undefined reference to puts`。那能不能写一个真正最小的、能直接在操作系统上跑起来的 Hello World？

可以。用汇编写，绕过 C 运行时。约定俗成的程序入口是 `_start`（不是 main），可以直接用 `ld` 链接。课程演示了一个最小版本：第一条指令就是自己写的代码，用 GDB 能从第一条指令开始单步调试。

但如果这个最小程序的 `_start` 直接 return 会发生什么？会 crash（segmentation fault）。原因要从状态机模型理解：C 程序从 main 开始执行，main 返回后栈空了，没有 topmost frame，取不出 PC，进入未定义状态。在真实的硬件上，进程初始化时栈上有一个初始的 return address，`_start` 执行 return（pop PC）时，从栈上 pop 出来的值是 0，PC 变成 0，访问地址 0 那一页是非法的，于是 segfault。

这说明真实的程序不是从 main 开始的。System V ABI 规定了整个进程的初始状态：加载时会跳转到一段初始化代码（crt 对象文件里的 `_start`），这段代码做好准备工作后调用 main，main 返回后这段代码继续执行，最终调用 exit 系统调用——而不是直接 return。

## 系统调用：程序与操作系统的唯一接口

这就引出了这节课的核心：系统调用（syscall）。C 语言本身没有"退出"这件事，程序要退出必须请求操作系统。操作系统和应用程序之间约定了一条特殊的指令来实现这个请求：x86 上是 `syscall`（64 位）或 `int 0x80`（32 位），ARM 上是 `svc`，RISC-V 上是 `ecall`。

要调用系统调用，把正确的参数放到正确的寄存器里：x86 上系统调用编号放 RAX，后续参数依次放 RDI、RSI、RDX；ARM 上编号放 X8，参数放 X0、X1、X2。比如要写一个最小的 Hello World：把文件描述符（1，标准输出）放 X0，字符串地址放 X1，字符串长度放 X2，write 系统调用的编号放 X8，执行 `svc`，Hello World 就打出来了。再执行 exit 系统调用（编号 + 返回值放 X0），程序就退出了。

```asm
// ARM64 最小 Hello World（示意）
_start:
    mov x0, #1          // stdout
    ldr x1, =msg        // 字符串地址
    mov x2, #12         // 长度
    mov x8, #64         // write 系统调用编号
    svc #0              // 触发系统调用

    mov x0, #1          // exit 返回值
    mov x8, #93         // exit 系统调用编号
    svc #0              // 程序终止
```

`svc` 执行后，控制权完全交给操作系统，后面的执行对程序来说是不可见的。操作系统根据寄存器里的参数（事先签好的"contract"）决定做什么：检查权限，合理就执行，不合理就返回失败。read 系统调用会在返回时把你请求的 buffer 填上数据；exit 系统调用则永远不会返回——程序的所有资源都被回收，进程消失。

一个生动的比喻：syscall 像是全身麻醉做手术。你先和医生（操作系统）签好 contract 说要做什么，然后闭眼（执行 syscall），醒来时手术可能做完了，也可能你变成了另一个人，也可能你再也醒不过来（exit）。

这是整个操作系统课最重要的一页：它回答了"什么是程序"——程序就是状态机的执行；回答了"什么是编译正确性"——输出序列相同；回答了"什么是操作系统"——就是一组通过 syscall 暴露的 API。除了 syscall 之外，所有东西都可以优化，只要 syscall 序列不变。

## 应用生态的层次

操作系统的所有程序和最小 Hello World 没有本质区别——都共享同一套 syscall。你看到的 VS Code、Cursor、Firefox、OBS、桌面环境，在操作系统面前都是平等的，都是通过 syscall 和操作系统交互的普通进程。

应用生态是分层的。最底层是直接使用 syscall 独立工作的 coreutils：ls、cp、mv、cat、grep 这些基本命令行工具。coreutils 之上是各种 binary utilities：as（汇编器）、ld（链接器）、objdump、size、strings 等，计算机系统基础课上用得很多。再往上是后台运行的 daemon：蓝牙管理（bluetoothd）、定时任务（cron）、U 盘挂载（udisksd）——你插上 U 盘桌面自动出现图标，就是 udisksd 这个 daemon 在后台通过 syscall 和操作系统通信后完成的工作。daemon 之上才是我们日常使用的 Application。

图形界面程序也是一样。XEdit 或 VS Code 不会直接往屏幕画像素——如果每个程序都直接占用整个屏幕，就没法管理了。它们通过工具库（如 GDK）向窗口管理器（如 Sway）发消息，Sway 再和底层的图形系统（如 Wayland）通信，Wayland 最终和操作系统内核交互完成渲染。这是一层一层抽象出来的，年复一年，有些软件被创造出来，有些死掉，最终到了今天这个任何应用开发都很简单的状态。

## strace：看见系统调用

如何证明"所有程序都和 minimum.s 一样"？用 strace。strace（system call trace）能打印出程序运行过程中的所有系统调用序列。加 `-f` 选项还能追踪子进程。

用 strace 追踪 `gcc a.c` 的编译过程，能看到 2249 个系统调用：读写文件、创建临时 `.o` 文件、execve 启动汇编器和链接器、unlink 删除临时文件。这证明了 GCC 编译确实经过了编译、汇编、链接这些步骤，而且是通过启动外部进程（execve 系统调用）实现的，而不是 GCC 自己在内存里完成所有事情。

在 AI 出现之前，分析这 2249 行 strace 输出靠的是 Unix philosophy：用管道把文本传给 grep，过滤掉不关心的部分，再传给下一个工具加工格式。比如过滤掉所有文件操作后剩 1700 行，再 grep `execve` 就能看到 GCC 在哪些路径下找汇编器、找到后启动它。

今天不需要这样了。直接让 AI 分析 strace 输出，它会自动写脚本、调用工具、理解 Unix philosophy，把进程的父子关系、每个阶段用的系统调用、完整的调用流程都画出来——比人在黑板上讲 5 分钟讲得还好。

## 二进制逆向与 AI 时代的工具思维

课程还演示了二进制逆向：用 AI 配合命令行工具（hexdump、objdump）hack 一个二进制文件，找到 Hello World 字符串的位置、定位设置返回值的指令、把返回值从 1 改成 42。AI 甚至能读懂 ARM 机器码，找到正确的字节位置并修改。这放在几十年前，是你在游戏存档文件里把一个字节从 00 改成 01 就改变游戏进度的事情——第一次二进制逆向的体验。

"Curiosity is all you need"——只要你知道这个世界是由 syscall 上面一层一层的工具构建起来的，知道有哪些工具、它们组合起来能做什么，就能让 AI 带着你做。在 coding agent 的辅助下，初学者和顶尖人类在工具使用能力上的差距已经几乎为零。AI 懂得在正确的时候调用正确的工具，做得非常快，而且代码可以自由组合无限复用。

比起教具体的概念（什么是 syscall、什么是 file descriptor），更重要的是建立用工具的思维。这门课最后会用一整个学期把 System Call 讲清楚——从应用视角看，操作系统就是这组 API，理解了它就理解了应用世界的一切是如何构建的。
